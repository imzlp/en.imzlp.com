Ultimate Optimization: Reducing the size of UE Android APKs

极致优化UE Android APK的大小
Posted on 2025-02-25 09:44 In UnrealEngine Views:
Symbols count in article: 13k Reading time ≈ 32 mins.

In game projects, when we package for various platforms, we always hope that the package for each platform can be minimized for easier distribution. Additionally, there are clear size requirements for publishing on certain platforms.

For UE, it contains a massive amount of code and a large number of plugins. The Build phase also generates reflection glue code, resulting in a significant increase in the amount of code generated during compilation. For example, on the Android platform, this leads to a considerable growth in the size of libUE4.so, putting pressure on both the package size and runtime memory.

Moreover, some necessary and extra resources brought in by the engine can also take up hundreds of MB, making the size of an empty APK easily reach several hundred MB! Not only to meet the requirements of the publishing platform but also from the perspective of package size and memory optimization, it is necessary to trim the size of the UE package.

This article will take Android as an example and introduce optimization ideas and practices for the trimmable parts in the UE package from various aspects while optimizing APK size and the runtime memory usage of native libraries. The strategies can also be reused on other platforms.

Package Size Distribution

In the APK, the part related to the game takes up a large portion of the space, mainly due to the following items:

  1. Executable code (so - lib/arm64-v8a)
  2. main.obb.png (game resources Pak, part of DirectoriesToAlwaysStageAsNonUFS)
  3. Files copied into the APK from third-party components

Specific optimization strategies need to be formulated for each of the three situations listed above.

Compress NativeLibs

When the APK is installed, there are two handling methods for NativeLibs:

  1. Decompress so into the application’s internal storage directory (/data/app/<package_name>/lib/) upon installation
  2. Load so directly from the APK file, which can speed up the installation process

Code Size Optimization

Regarding the code size optimization part, for the Android platform, the core goal is to reduce the size of individual so files! Furthermore, we aim to minimize any impact on runtime performance.

Optimization ideas for the size of NativeLibs:

  1. Reduce the number of dynamic link libraries, removing unnecessary ones
  2. Reduce internal symbols and code segment sizes
  3. Remove debugging information

For all so files, these optimization strategies can be applied at compile/link time. However, for UE projects, we usually only have control over the engine and project code. The library code needs to be optimized by the library provider. Therefore, the subsequent optimization strategies apply only to libUE4.so/libUnreal.so.

Reduce libUE4.so

During packaging, since a complete compilation is needed, and UE operates in Monolithic mode by default at runtime, all code is compiled into the same executable file. (Previous articles have detailed this: UE Plugin and Tool Development: Basic Concepts)

The compilation process based on UBT packages, along with the configuration parameters provided in target.cs/build.cs, allows us to control compilation of the engine and project code to achieve our aim of minimizing so size.

For UE projects, there are several ideas for optimizing so size:

  1. Disable unnecessary modules
  2. Control code optimization (control inline/O3/Oz)
  3. Disable unnecessary exception handling for Modules
  4. Enable LTO
  5. Remove unnecessary exported symbols

Disable Modules

You can disable modules that are clearly not needed in the engine within target.cs:

At the same time, need to sort out unnecessary runtime plugins introduced in the project to reduce the number of Modules participating in the compilation, thereby minimizing the actual code compiled.

Disable Inline

Inline is a compile-stage optimization that directly replaces function calls with function code rather than conventional function calls. It can reduce the overhead of function calls and theoretically improve program execution efficiency.

However, inlining can increase the size of the .text section, so it can be disabled as appropriate.

  1. Modify target.cs: bUseInlining = false; (Effective only on IOS/Linux/Mac/Win)
  2. Modify UBT and control it during Android compilation with bUseInlining, adding the -fno-inline-functions compile parameter

Note: Disabling inline may result in some performance loss for frequently called functions. In non-frequent cases, the performance difference between inlined and non-inlined can be quite minimal. In my test results, whether inlined had a negligible impact on frame rates.

Disable Exception Handling

Some modules have C++ exception handling enabled, but no try/catch is being utilized:

This can be turned off, which reduces the size of .eh_frame in so.

Use O3/Oz Compilation

Control the value of bCompileForSize in target.cs, which can choose to compile with O3 or Oz:

The difference between O3 and Oz:

  • -O3: Performance first, aggressively inlines, unrolls loops
  • -Oz: Size first, avoids inlining, conserves loops

Choose the approach based on the project’s actual performance conditions.

Enable LTO

LTO stands for Link Time Optimization, which can eliminate dead code at link-time, optimize cross-module function calls, inline, etc. In the engine’s build.cs, bAllowLTCG can be enabled, with LTCG being a form of LTO, but it is only effective on IOS/Linux/Mac/Win (UE4.25).

For Android support, adjustments must be made to UBT (AndroidToolChain.cs) to control whether to add the -flto=thin compile parameter based on bAllowLTCG. thin is a version that strikes a balance between size reduction and optimization time.

Remove Exported Symbols

By default, unless specified otherwise, all functions and variables are exported during the compilation of so, allowing access from other so files. However, in the UE engine, only a rare few interfaces are explicitly accessible externally (related to JNI), so the symbol exports of libUE4.so are mostly wasted. Removing symbol exports can significantly reduce the size of so and memory usage!

Modern compilers provide control mechanisms for exported symbols at link-time through version-script, which can control symbol behavior during linking by passing an ldscript file.

During the compilation process, a ldscript file must first be constructed, filled with the symbol export control code, and then passed to the linker in target.cs:

For UE, only the following three types of matching symbols should be exported: Java_*/ANativeActivity_onCreate/JNI_OnLoad; all others can be removed.

Optimize Data

After the series of code size optimizations described above, the benefits become clear.

Size of so after Compression

As mentioned earlier, NativeLibs can be compressed when entering the APK; thus, reducing the original size of so also reduces the size after compression.

After the optimizations discussed above, the original size of the so in Shipping mode was reduced from 258M to 146M!
The compressed size of the so reduced from 74.3M to 44.67M, a decrease of 29.63M! The executable file has significantly shrunk.

Resource Trimming

Files within the APK

There are some third-party plugins that copy files into the APK, which can also be optimized.

It is necessary to analyze the actual usage status of the project:

  1. Remove unnecessary third-party components
  2. For necessary components, remove unnecessary files

Component Trimming: Taking GVoice as an Example

If the project integrates the GCloud component, it copies components to the APK file. The model files of GCloudVoice take up a significant portion.

The assets/GCloudVoice directory in the APK compresses to about 13.5M:

  • wave_dafx_data.bin is for 3D voice; if 3D functionality is not used, it can be removed.
  • wave_3d_data.bin is for 3D voice; if 3D functionality is not used, it can be removed.
  • cldnn_spkvector.mnn is for voiceprint extraction; if this feature is not used by default, it can be removed.
  • libwxvoiceembed.bin is for civilized speech; if civilized speech is not required, it can be removed.
  • libgvoicensmodel.bin is the noise suppression algorithm model and cannot be deleted.
  • decoder_v4_small.nn and encoder_v4_small.nn are used for aicodec; if aicodec is not used, they can be deleted.
  • dse_v1.nn, dse_v1_align.nn, dse_v1_mono.nn are resources for a new algorithm under wwise; if there are packaging size restrictions, they can also be removed.

Unused model files can be excluded from the project. Additionally, it is preferable to not delete files directly from an implementation perspective but to modify the copy logic in GVoice_APL.xml:

In-Game Resources

In-game resources are the resources/files that the UE engine or components depend on, packaged into PAK or copied into main.obb files.

  • In PAK: The game assets must be analyzed for which are non-essential, which can be trimmed, or which can be loaded lazily.
  • DirectoriesToAlwaysStageAsNonUFS: Not included in PAK but will be packaged into the main.obb.

PAK Internal Resources

More accurately described as: the resources within the installation package PAK.

The engine’s necessary resources are all in pakchunk0. Besides pakchunk0, UE can package chunks split using PrimaryAssetLabel into the installation package.

However, even with the resources or files in pakchunk0, further optimization is still needed:

  1. Retain only engine-essential resources (key assets in /Engine, ini, GlobalShader, project ShaderLibrary, startup map, GameFramework assets, etc.). I previously detailed this in UE Resource Management: Analysis of Engine Packaging Resources.
  2. Remove non-essential resources that are not needed during the initialization phase.
  3. Refactor parts of the engine for lazy loading of certain files (such as loading L10N localized languages).
  4. Split startup phase resources from in-game resources (like fonts), packaging in-game fonts separately for dynamic download.

The engine’s unpacking logic also has significant limitations, similar to ShaderLibrary, which generates a single large resource for the entire project. When the scale is large, it can become a bottleneck for optimizing package size. Detailed explanations of this content can be found in my previous article (Resource Management: Restructuring UE’s Package Splitting Scheme).

Moreover, all Android resources within the installation package must be excluded during resource management and packaging, transforming them into a dynamic download/mounting mechanism that does not affect iOS. When using schemes such as splitting pak with PrimaryAssetLabel, the engine provides a method for Android to exclude Paks from the installation package:

However, official support is only provided for the Android platform, making it less convenient for other platforms. In a previous article, I mentioned that the HotChunker extension I developed can easily realize a universal package filtering scheme, supporting a customizable package control strategy across all platforms.

StageAsNonUFS

In the engine’s packaging configuration, there is a DirectoriesToAlwaysStageAsNonUFS option, which specifies directories that do not pack into PAK but will pack into main.obb. Currently, only the Content/Movies directory is copied into main.obb.

Moreover, the Ini read during packaging has a hierarchical logic, allowing differentiation in configurations for different platforms during packaging! If differentiation between Android/iOS is desired, this mechanism can also be utilized.

The packaging strategy can be adjusted as follows: unless necessary videos (such as those that need to play immediately on startup), the remaining in-game MP4s can be packaged into PAK separately for dynamic download.

This can significantly reduce the size of MP4s within the APK and also allow for hot updates.

Optimization Results

Through the various package size optimization methods discussed above, the game’s APK size was successfully reduced from several hundred MB to around a hundred MB while retaining complete third-party components, game functionalities, and allowing for resources to be dynamically downloaded, making the installation package itself a very small downloader that is easy to distribute.

The specific optimization strategies employed should be based on the actual needs of the project and balance between package size and performance, such as controlling inline and compile optimization levels, extreme refinement of resources (like L10N), which may also require modifications to the engine, etc.
